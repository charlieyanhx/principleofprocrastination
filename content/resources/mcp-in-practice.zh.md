一个采购智能体需要在WMS中查库存、在ERP中校验供应商报价、在MES中更新采购单状态——所有操作在一个决策周期内完成。如果没有统一的工具接口协议，每个集成都是一个定制适配器，有各自的认证方式、错误语义和数据格式。六个企业系统乘以三种智能体类型，就是一个没有团队能长期维护的集成面。

模型上下文协议（MCP）解决的就是这个问题：标准化LLM智能体如何发现、调用和接收外部工具的结果。它不替代你的企业API，它在API外面包了一层统一接口，让任何MCP兼容的智能体都能消费，而不需要写系统专属的集成代码。

本文是面向企业系统的MCP工具接口实施指南。它建立在[多智能体协调模式](/resources/agent-coordination-patterns)中的信号-决策-执行架构和[工业场景中的LLM编排](/resources/llm-orchestration-industrial)中的编排原则之上。如果那些文章解释了智能体系统*为什么*需要结构化工具接口，这篇文章讲的是*怎么*构建它们。

## MCP到底是什么

MCP是基于JSON-RPC 2.0的协议，定义了三个原语：**资源**（智能体可读的数据）、**提示**（模板化工作流）和**工具**（智能体可调用的函数）。对于企业集成来说，工具是核心原语。

架构分三个角色：

- **宿主（Host）**：LLM应用（你的智能体运行时）
- **客户端（Client）**：宿主内部管理MCP会话的连接器
- **服务端（Server）**：暴露工具的服务，工具包装了你的企业系统API

服务端通过`tools/list`广播可用工具及其输入输出模式。客户端发现工具后，根据智能体推理选择合适工具，通过`tools/call`调用。结果以结构化内容（JSON）或非结构化内容（文本、图片）返回。

MCP不是智能体框架，不是API网关，也不是工作流引擎。它是一个接口协议。用友的OpenAPI还是用友的OpenAPI，MCP提供的是LLM与那个API之间的类型化契约。

## 工具模式设计

工具定义是企业MCP实施成败的关键。一个设计糟糕的工具模式会让LLM猜测参数格式、产生歧义结果、引发下游连锁错误。

每个MCP工具定义有四个组件：`name`、`description`、`inputSchema`，以及可选的`outputSchema`。以下是一个用于从ERP查询采购单状态的生产级工具：

```json
{
  "name": "erp.purchase_order.get_status",
  "description": "根据采购单号查询采购单当前状态和审批链。返回订单状态、行项目、审批历史和阻塞异常。当你需要检查采购单是否已审批、待审核或已驳回时使用此工具。",
  "inputSchema": {
    "type": "object",
    "properties": {
      "po_number": {
        "type": "string",
        "description": "采购单号，格式为 PO-YYYYMMDD-XXXX",
        "pattern": "^PO-\\d{8}-\\d{4}$"
      },
      "include_line_items": {
        "type": "boolean",
        "description": "是否包含行项目明细。仅查状态时设为false可减少返回数据量。",
        "default": false
      }
    },
    "required": ["po_number"],
    "additionalProperties": false
  },
  "outputSchema": {
    "type": "object",
    "properties": {
      "po_number": { "type": "string" },
      "status": {
        "type": "string",
        "enum": ["draft", "pending_approval", "approved", "rejected", "cancelled", "received"]
      },
      "current_approver": { "type": "string" },
      "total_amount": { "type": "number" },
      "currency": { "type": "string" },
      "blocking_exceptions": {
        "type": "array",
        "items": { "type": "string" }
      }
    },
    "required": ["po_number", "status"]
  }
}
```

几个关键设计决策：

**命名空间化的工具名。** `erp.purchase_order.get_status` 而不是 `getPOStatus`。当智能体同时接入40+个工具横跨ERP、MES、WMS时，命名空间结构决定了工具选择是可靠的还是随机的。

**引导LLM的描述。** 描述不是给人看的文档——它是LLM决定是否调用此工具的首要信号。"当你需要检查采购单是否已审批时使用此工具"比"获取采购单状态"更有用。

**受约束的输入模式。** `po_number`上的`pattern`字段阻止LLM把供应商名称当采购单号传入。每个有已知格式的参数都应声明约束。枚举、正则、最大最小值——这些不是锦上添花，是护栏。

**结构化输出模式。** 2025年11月MCP规范引入了`outputSchema`和`structuredContent`。对于企业集成，务必定义输出模式。这让LLM能程序化解析结果，而不是从自由文本中提取数据。

**`additionalProperties: false`。** 阻止LLM自行发明参数。不加这个，你会看到模型自信地传入`urgent: true`这样你的API默默忽略的字段。

### 工具粒度

一个常见错误是把工具设计成企业系统API的一对一映射。一个ERP可能暴露200+个REST端点。暴露200个MCP工具只会让模型选择混乱。

围绕智能体任务设计工具，而非系统能力：

- **粒度太细**：`erp.item.get`、`erp.item.get_price`、`erp.item.get_stock`、`erp.item.get_supplier`
- **粒度太粗**：`erp.query`（接受自然语言问题）
- **恰当粒度**：`erp.item.lookup`（一次调用返回物料详情、当前库存、价格和主供应商）

恰当的粒度取决于智能体的决策拓扑。信号智能体聚合库存状态，需要`wms.inventory.get_levels`一次返回多仓库库存。决策智能体路由审批，需要`erp.purchase_order.get_status`和`erp.purchase_order.approve`分开——因为读和写是不同的授权范围和不同的审计事件。

## 认证模式

2025年11月MCP规范引入了完整的OAuth 2.1授权框架。企业系统从不允许匿名访问，这部分值得详细了解。

### 传输层认证

MCP支持两种传输机制，认证模式不同：

**stdio传输**（本地进程）：MCP服务端作为子进程运行。认证继承自环境——进程从环境变量、配置文件或密钥管理器读取凭据。线上没有令牌交换。适用于智能体运行时和MCP服务端共享信任边界的单租户部署。

**HTTP传输**（远程服务）：MCP服务端是网络服务，OAuth 2.1生效。流程：

1. 客户端发送未认证请求
2. 服务端返回`401 Unauthorized`，`WWW-Authenticate`头包含受保护资源元数据URL
3. 客户端通过RFC 9728发现授权服务器
4. 客户端执行OAuth 2.1授权（始终使用PKCE）
5. 每个后续请求携带`Bearer`令牌

### 企业认证拓扑

实际场景中，MCP服务端很少自己管认证。企业系统有现成的身份提供商——Active Directory、企业微信SSO、或ERP内置认证（如用友YonBIP的统一身份认证、金蝶苍穹的IAM）。MCP服务端作为OAuth 2.1资源服务器，把认证委托给企业IdP。

典型模式：

```
智能体运行时（MCP客户端）
  → MCP服务端（资源服务器，验证令牌）
    → 企业IdP（授权服务器，发放令牌）
      → ERP/MES/WMS API（接受服务间凭据）
```

MCP服务端向IdP验证智能体令牌，然后用独立的服务账号或客户端凭据授权调用企业API。智能体的令牌和ERP的令牌永远不是同一个令牌。规范明确要求：MCP服务端不得将从MCP客户端收到的令牌直接传递给上游API。

### 基于范围的工具访问

MCP的授权框架支持增量范围协商。这自然映射到企业权限模型：

- `erp:po:read` — 查询采购单状态
- `erp:po:write` — 审批或驳回采购单
- `wms:inventory:read` — 查库存
- `mes:schedule:write` — 调整生产排程

当智能体尝试调用无权限的工具时，服务端返回`403 Forbidden`加`error="insufficient_scope"`及所需范围。智能体运行时可触发步进授权流程——请求额外范围，必要时获取用户同意，然后重试。

在工业部署中，信号智能体默认持有只读范围。如果遇到需要决策智能体审批采购的异常，决策智能体的工具调用携带写入范围。范围边界强制分离了观察与行动。

## 让LLM能自我纠正的错误处理

MCP区分两类错误，这个区分对企业集成至关重要：

**协议错误**（JSON-RPC错误响应）：请求本身格式错误——未知工具名、无效JSON、服务端崩溃。返回标准JSON-RPC错误码（-32600无效请求、-32601方法未找到、-32602无效参数）。LLM通常无法从这类错误自我纠正。

**工具执行错误**（结果中`isError: true`）：工具执行了但操作失败——采购单号不存在、库存API超时、审批被业务规则拒绝。以正常结果结构返回，`isError: true`加文本消息。LLM*能*从这类错误自我纠正。

企业系统的实现模式：

```json
{
  "jsonrpc": "2.0",
  "id": 42,
  "result": {
    "content": [
      {
        "type": "text",
        "text": "采购单PO-20260401-0127未找到。单号格式正确但无匹配记录。请与申请人确认采购单号，或使用erp.purchase_order.search按供应商名称搜索。"
      }
    ],
    "isError": true
  }
}
```

错误消息做了三件事：(1)确认尝试了什么，(2)解释为什么失败，(3)建议替代行动。这不只是良好实践——它决定了智能体是重试同一个失败调用五次，还是转向基于搜索的方法。

### 企业场景错误分类

为每个企业系统集成定义错误类别，映射到可执行的响应：

| 错误类别 | 示例 | 智能体响应 |
|---|---|---|
| 未找到 | 采购单号不存在 | 用替代标识符搜索 |
| 权限拒绝 | 用户无审批权限 | 升级到有权审批人 |
| 校验失败 | 金额超过审批额度 | 拆分采购单或升级 |
| 系统不可用 | ERP维护窗口 | 排队等待并退避重试 |
| 数据过期 | 库存计数已过时 | 从源系统刷新 |
| 业务规则违反 | 供应商在冻结名单 | 标记异常交人工处理 |

每个类别都应产生引导LLM走向正确恢复路径的错误消息。笼统的"操作失败"毫无用处——只会产生重试循环。

### 熔断器模式

企业系统有维护窗口、性能退化和偶尔的宕机。MCP服务端应对每个上游系统实现熔断器：

- **关闭**（正常）：请求通过
- **打开**（系统宕机）：立即失败，返回明确错误（"ERP系统正在计划维护中，预计06:00恢复。当前无法执行采购单操作。"）
- **半开**（探测）：放行一个请求测试恢复

熔断器状态应体现在工具错误消息中，让LLM知道不要重试。一个持续敲击已宕机ERP的智能体不仅浪费资源——还可能触发限流影响其他系统。

## 企业系统集成细节

### ERP集成（用友、金蝶、SAP、Oracle）

ERP是最复杂的MCP集成目标。深度对象模型、事务语义、跨实体的业务规则。

金蝶云·苍穹已宣布全面支持MCP协议标准，将所有SaaS Open API封装为标准MCP服务，覆盖订单管理、库存管理、财务结算等数千个API。用友YonBIP平台开放2万+API接口，依托YonGPT企业服务大模型已开发上千个智能集成智能体。这意味着国内ERP厂商正在主动拥抱MCP，但自建MCP服务端仍然是常见需求——因为你需要按智能体的决策拓扑组织工具，而非按ERP的API结构。

关键模式：

**以读为主的工具设计。** 大多数智能体与ERP的交互是读操作：查采购单状态、查价格、核实供应商详情。读工具应返回反范式化视图——智能体不应该需要五次调用才能组装一个决策所需的上下文。

**事务性写入。** ERP写操作（创建采购单、过账收货）是事务性的。MCP服务端必须处理ERP的事务生命周期：开启事务、校验、提交或回滚。不要把部分写入暴露为独立工具——`create_purchase_order`工具应原子性地处理行项目，而非要求单独的`add_line_item`调用。

**SAP的RFC/BAPI包装。** SAP的原生接口使用BAPI和RFC。MCP服务端将这些包装在干净的工具模式后面，在SAP独特的参数命名和合理的JSON属性之间做转换。

### MES集成

制造执行系统的运行节奏比ERP快——生产工单、工位状态、质量冻结以分钟为单位变化。

**实时读取模式。** MES工具应返回当前状态，不是缓存状态。`mes.work_order.get_status`应直接调MES API，而非读同步表。

**写操作的护栏。** 生产排程变更可能停掉一条产线。MES写工具需要额外的确认机制：工具应返回影响预览（"重排工单WO-4521将导致下游工单WO-4522和WO-4523延迟2小时"），并要求明确确认后再执行。

### WMS集成

仓储管理系统提供库存可见性和拣选/包装/发货流程控制。

**多仓位聚合。** 有用的WMS工具在一次响应中返回所有相关仓位的库存，包括预留库存、在途数量和可承诺量。暴露逐仓位查询会迫使智能体做N次调用再聚合——这是MCP服务端应该处理的任务。

**批次和序列号处理。** 受监管行业（医药、食品、航空）要求批次可追溯。WMS工具的输入输出都必须包含批次/序列号信息。一个不含批次数据的`wms.inventory.get_levels`工具，对质量冻结决策毫无用处。

## 部署架构

生产环境中企业集成的MCP部署通常如下：

```
┌─────────────────────────────────────┐
│         智能体运行时（宿主）            │
│  ┌─────────┐  ┌─────────┐          │
│  │ 信号    │  │ 决策    │          │
│  │ 智能体    │  │ 智能体    │          │
│  └────┬────┘  └────┬────┘          │
│       │            │               │
│  ┌────┴────────────┴────┐          │
│  │     MCP客户端         │          │
│  └──────────┬───────────┘          │
└─────────────┼───────────────────────┘
              │ JSON-RPC / HTTP+SSE
              │
┌─────────────┼───────────────────────┐
│  MCP网关（负载均衡）                  │
│  - 限流                             │
│  - 令牌验证                          │
│  - 请求路由                          │
└──────┬──────────┬──────────┬────────┘
       │          │          │
┌──────┴───┐ ┌───┴─────┐ ┌─┴────────┐
│ ERP MCP  │ │ MES MCP │ │ WMS MCP  │
│ 服务端    │ │ 服务端   │ │ 服务端    │
│          │ │         │ │          │
│用友/金蝶  │ │ REST API│ │ REST API │
└──────┬───┘ └───┬─────┘ └┬─────────┘
       │         │        │
  [ERP系统]   [MES]    [WMS]
```

每个企业系统有自己的MCP服务端。MCP网关处理横切关注点：认证、限流、审计日志。智能体运行时的MCP客户端连接网关，网关将`tools/call`请求路由到对应的后端服务端。

这种分离对运维意义重大：

- **独立扩缩。** WMS MCP服务端在仓库高峰期可能需要10倍于ERP服务端的请求容量。
- **独立生命周期。** 更新MES MCP服务端不需要动ERP集成。
- **爆炸半径控制。** 用友连接断了，MES和WMS工具依然可用。

## 这能带来什么

良好实施的MCP层在智能体和企业系统之间产生三个可衡量的结果：

**集成速度。** 添加新工具需要几小时，不是几周。模式契约是标准化的——新的`erp.supplier.lookup`工具遵循与所有其他工具相同的模式。没有新的适配器代码，没有自定义序列化，没有新颖的错误处理。

**智能体可靠性。** 类型化输入模式防止参数混淆。输出模式实现确定性结果解析。错误消息引导自我纠正。LLM把上下文窗口用在推理上，而不是用在摸索API怪癖上。

**运营治理。** 每次工具调用都是一个离散的、有日志的事件，有清晰的模式。你可以审计智能体从ERP读了什么、传了什么参数、收到了什么结果。结合基于范围的授权模型，你拥有了企业IT所要求的控制面。

MCP不是企业AI集成的终极答案——协议仍在成熟中，2025年11月规范是向生产就绪迈出的重要一步但不是最后一步。它提供的是一个理性的起点：LLM推理与企业系统操作之间的类型化、认证的、可审计的接口。对于需要让智能体系统上线的团队来说，这足以开始了。
